# Workflow Engine

## Overview

A three-phase lifecycle for multi-step AI task automation:

1. **Generate**: User describes what they want in natural language → `claude -p` brainstorms a workflow definition
2. **Edit**: User reviews/refines the generated JSON in the dashboard
3. **Compile**: `claude -p` "compiles" the source JSON into deterministic executable JS (conditions become code, prompts become builders)
4. **Run / Test**: Engine walks compiled steps, each step being a full Claude Code `-p` session. Testable directly from the GUI.

## Core Principle: Each Step = Full Claude Code Session

A workflow step is NOT a single LLM call. Each step spawns `claude -p` — the same agentic loop users get in the Claude tab. When a step says "Fix the security issues", Claude may make 15+ tool calls: read files, edit code, run tests, retry. The `-p` flag gives it the complete agent loop with tool use, multi-turn reasoning, file access, and shell commands.

**Implications:**

- **No "shell" step type needed** — Claude Code runs shell commands as tools within its own agent loop
- **The runtime engine is a session orchestrator, not an LLM orchestrator.** Per step: spawn `claude -p` → stream events → capture final result → pass to next step
- **`buildPrompt(ctx)` is just the initial user message** — same as what you'd type in the Claude tab
- **`parseOutput(raw)` is lightweight** — operates on the `result` event after Claude's full agent loop
- **Context passing is the real value** — the compiled workflow's main job is building prompts that inject previous steps' outputs
- **Each step is a fresh session** — no `--resume` between steps, because steps can use different profiles
- **Steps route through the proxy** — every API call within a step shows up in the Inspector tab
- **Streaming per step** — the same `stream-json` events that drive the Claude tab

## Dependency: Multi-Claude Tabs

Workflows spawn steps as Claude tab instances via the `ClaudeSessionManager` (see `MULTI_CLAUDE_TABS.md`). Each step gets a `tabId` like `wf-<runId>-<stepId>`. Parallel steps run as parallel tab instances.

## Source Format (.workflow.json)

This is what the user edits. Natural language everywhere — no code required.

```json
{
  "name": "review-and-fix",
  "description": "Review code for issues and fix them",
  "workingDirectory": "/home/hp/myproject",
  "inputs": {
    "target": "file or directory to review",
    "focus": "what kind of issues to look for"
  },
  "steps": {
    "analyze": {
      "profile": "deep-review",
      "do": "Analyze {{target}} for {{focus}} issues",
      "produces": "list of findings with file, line, severity"
    },
    "decide": {
      "condition": "there are findings from analyze",
      "then": "fix",
      "else": "done"
    },
    "fix": {
      "profile": "fast-coder",
      "do": "Fix the issues found",
      "context": ["analyze"],
      "next": "test"
    },
    "test": {
      "profile": "test-runner",
      "do": "Run the test suite and verify fixes",
      "condition": "all tests pass",
      "then": "done",
      "else": "fix",
      "maxRetries": 3
    },
    "done": {
      "do": "Summarize what was done"
    }
  }
}
```

### Step Fields

| Field | Type | Description |
|-------|------|-------------|
| `profile` | string | Which profile to use (model, permissions, budget) |
| `do` | string | Natural language instruction — the "user message" for `claude -p` |
| `produces` | string | Describes expected output shape (helps compilation) |
| `context` | string[] | Step IDs whose output to include in the prompt |
| `condition` | string | Natural language condition for branching |
| `then` / `else` | string | Branch targets when condition is present |
| `next` | string | Explicit next step (when no condition) |
| `maxRetries` | number | Retry loop limit for condition-based retry |
| `parallel` | string[] | Step IDs to run concurrently (fan-out) |
| `join` | string | Step to continue after parallel steps complete (fan-in) |

## Compiled Format (.compiled.js)

Generated by `claude -p` from the source JSON. Deterministic — no AI calls for routing.

```javascript
module.exports = {
  name: "review-and-fix",
  sourceHash: "a3f8c2...",
  inputs: {
    target: { type: "string", required: true, description: "file or directory" },
    focus: { type: "string", default: "security, performance" },
  },
  steps: [
    {
      id: "analyze",
      profile: "deep-review",
      buildPrompt(ctx) {
        return `Analyze ${ctx.inputs.target} for ${ctx.inputs.focus} issues.\n\n`
          + `Output as JSON array: [{file, line, severity, description}]`;
      },
      parseOutput(raw) {
        const m = raw.match(/\[[\s\S]*\]/);
        return m ? JSON.parse(m[0]) : [];
      },
      next: "decide",
    },
    {
      id: "decide",
      type: "condition",
      evaluate(ctx) {
        const f = ctx.steps.analyze?.output;
        return Array.isArray(f) && f.length > 0;
      },
      then: "fix",
      else: "done",
    },
    // ... remaining steps
  ],
};
```

### What Compilation Does

- Natural language conditions → JS predicates (no AI at runtime for routing)
- `do` text → `buildPrompt(ctx)` with context injection and output format instructions
- `produces` → `parseOutput(raw)` extractors
- Data flow wired explicitly via `ctx.steps[id].output`

## Three-Phase Generation

### Phase 1: Generate (creative)

User types a description → `claude -p` brainstorms the workflow structure → returns a `.workflow.json` → dashboard shows it in the editor modal.

**Redo**: Previous generation + user feedback sent to a new `claude -p` call.

### Phase 2: Edit (manual)

User edits JSON in the overlay modal editor. Saves → written to `capabilities/workflows/<name>/workflow.json`.

### Phase 3: Compile (mechanical)

User clicks Compile → `claude -p` transforms to deterministic JS → writes `compiled.js` → list shows "compiled" badge.

**Recompile detection**: `compiled.js` stores `sourceHash`. If source changes, badge shows "needs recompile".

## Runtime Engine

State machine in `src/workflows.js`. Each agent step spawns a full `claude -p` session.

```
loadCompiled(name) → steps[]
initContext(inputs) → ctx

for each step:
  if condition step:
    evaluate(ctx) → follow then/else branch
  if agent step:
    prompt = buildPrompt(ctx)
    profile = loadProfile(step.profile)
    args = buildClaudeArgs(profile)
    tabId = `wf-${runId}-${step.id}`
    spawn claude -p with args, pipe prompt to stdin
    stream events → broadcast as workflow:step:progress
    on result event → parseOutput(raw) → ctx.steps[step.id].output
    evaluate next/condition → determine next step
  if parallel step:
    spawn multiple sessions simultaneously
    wait for all, collect outputs at join step

broadcast workflow:run:complete
save run to runs/
```

### Workflow Test Feature — Detailed Flow

#### Starting a Test

1. User clicks `▶` (test button) on a workflow row in the list
2. If workflow is not compiled: show error toast "Compile workflow first"
3. If compiled: open the **Test/Run Modal** (`#wfRunModal`)
4. Auto-generate input fields from `workflow.inputs`:
   - Each input key → `<label>key: <input type="text" placeholder="description"></label>` inside a `.pm-field-row`
   - Pre-fill with defaults if the input description suggests them
5. User fills in inputs, clicks **Run**

#### During Execution

1. **Run** button hides, **Stop** button appears
2. Status badge changes: `ready` → `running` (`.ref-tag.tag-ex` with pulse)
3. Step list appears with all steps in `pending` state
4. As each step executes:
   - Step row transitions: `pending` → `running` (icon pulses, accent color)
   - Stream-json events arrive as `workflow:step:progress` WS messages
   - If step's expand button is clicked, streaming text appears below the row in a `.wf-run-output` div (monospace, scrollable, max-height 300px)
   - On step completion: `running` → `done` (green) or `failed` (red), elapsed time shown
   - Condition steps resolve instantly: show branch taken (e.g., "→ fix")
5. If user clicks **Stop**: sends `workflow:run:cancel`, step goes to `failed`, run status → `cancelled`

#### After Completion

1. Status badge: `running` → `completed` (green) or `failed` (red)
2. **Stop** button hides, **Run** button reappears (for re-run)
3. All step rows show final status with elapsed times
4. User can expand any step to see its full output
5. Run results are saved to `capabilities/workflows/<name>/runs/run-<ts>.json`

### Frontend Module — `public/workflows.js`

IIFE following the `capabilities.js`/`mcp.js` pattern:

```javascript
(function workflowModule() {
  'use strict';
  const { state, sendWs, escHtml, jsonBlock } = window.dashboard;

  // Internal state
  const wf = {
    workflows: [],          // [{name, description, compiled, sourceHash, compiledHash}]
    generating: false,      // currently generating via claude -p
    compiling: null,        // name being compiled, or null
    lastGenerated: null,    // last generated JSON (for redo context)
    // Run state
    activeRunId: null,
    activeRunName: null,
    runStatus: 'ready',     // ready | running | completed | failed | cancelled
    runSteps: [],           // [{id, status, output, elapsed, profile, instruction}]
    runCurrentStep: null,
    expandedSteps: new Set(), // step IDs with visible output
  };

  // --- Message handling ---
  function handleMessage(msg) {
    switch (msg.type) {
      case 'workflow:list':
        wf.workflows = msg.workflows || [];
        renderList();
        break;
      case 'workflow:loaded':
        openEditModal(msg.name, msg.workflow);
        break;
      case 'workflow:generated':
        wf.generating = false;
        wf.lastGenerated = msg.workflow;
        openEditModal(null, msg.workflow, true); // true = show redo bar
        break;
      case 'workflow:compiled':
        wf.compiling = null;
        renderList();
        break;
      case 'workflow:error':
        wf.generating = false;
        wf.compiling = null;
        showError(msg.error);
        break;
      case 'workflow:run:started':
        wf.activeRunId = msg.runId;
        wf.runStatus = 'running';
        updateRunUI();
        break;
      case 'workflow:step:start':
        updateStepStatus(msg.stepId, 'running');
        break;
      case 'workflow:step:progress':
        appendStepOutput(msg.stepId, msg.text);
        break;
      case 'workflow:step:complete':
        updateStepStatus(msg.stepId, msg.success ? 'done' : 'failed', msg.elapsed);
        break;
      case 'workflow:run:complete':
        wf.runStatus = msg.status; // 'completed' or 'failed'
        updateRunUI();
        break;
    }
  }

  // --- Rendering ---
  function renderList() { /* rebuild #wfList using cap-list-item pattern */ }
  function openEditModal(name, workflow, showRedo) { /* populate and show #wfModal */ }
  function openRunModal(name) { /* populate inputs form and show #wfRunModal */ }
  function updateRunUI() { /* update status badge, button visibility */ }
  function updateStepStatus(stepId, status, elapsed) { /* update step row */ }
  function appendStepOutput(stepId, text) { /* append to step's output area */ }
  function showError(text) { /* brief error display */ }

  // --- Event handlers (delegated) ---
  // Generate button, New button, Edit/Delete/Test/Compile per-row buttons
  // Modal save/cancel/redo, Run modal start/stop/close
  // Step expand toggle

  window.workflowModule = { handleMessage };
})();
```

### WS Message Protocol

| Direction | Type | Payload | Purpose |
|-----------|------|---------|---------|
| C→S | `workflow:list` | — | Request workflow list |
| S→C | `workflow:list` | `{workflows}` | Return list with status |
| C→S | `workflow:load` | `{name}` | Load workflow source |
| S→C | `workflow:loaded` | `{name, workflow}` | Return source JSON |
| C→S | `workflow:save` | `{name, workflow}` | Save source JSON |
| C→S | `workflow:delete` | `{name}` | Delete workflow |
| C→S | `workflow:generate` | `{description, feedback?}` | Generate from NL |
| S→C | `workflow:generated` | `{workflow, raw}` | Return generated JSON |
| C→S | `workflow:compile` | `{name}` | Compile source → JS |
| S→C | `workflow:compiled` | `{name, success}` | Compilation result |
| S→C | `workflow:error` | `{error}` | Error message |
| C→S | `workflow:run` | `{name, inputs}` | Start a run |
| S→C | `workflow:run:started` | `{runId, name}` | Run started |
| S→C | `workflow:step:start` | `{runId, stepId}` | Step starting |
| S→C | `workflow:step:progress` | `{runId, stepId, text}` | Step streaming output |
| S→C | `workflow:step:complete` | `{runId, stepId, success, elapsed, output}` | Step finished |
| S→C | `workflow:run:complete` | `{runId, status}` | Run finished |
| C→S | `workflow:run:cancel` | `{runId}` | Cancel running workflow |

## MCP Integration

Register as MCP tools so any Claude session can invoke workflows:

- **`workflow_list()`** — list available workflows with status
- **`workflow_run(name, inputs)`** — execute a compiled workflow
- **`workflow_status(runId)`** — get current step, outputs so far

## Implementation Order

1. **Multi-Claude Tabs** (prerequisite) — see `MULTI_CLAUDE_TABS.md`
2. **Workflow backend**: CRUD + generate + compile in `src/workflows.js`
3. **Workflow runtime**: step execution via session manager
4. **Workflow frontend**: list, edit modal, test modal, step progress — all in `public/workflows.js`
5. **MCP tools**: expose `workflow_run` etc.

## Key Design Decisions

1. **Each step = full `claude -p` session** — the runtime is a session orchestrator. All tool use, retries, multi-turn reasoning happen inside Claude Code's agent loop.
2. **Fresh session per step, no --resume** — steps can use different profiles. Context passed explicitly via `buildPrompt(ctx)`.
3. **Generate + Compile are separate `claude -p` calls** — generation is creative, compilation is mechanical.
4. **Compiled.js is CommonJS** — project uses `require()`. Cache-bust with `delete require.cache[path]`.
5. **Profile reuse** — steps reference profiles by name, resolved via `caps.loadProfile()`.
6. **Stream-json parsing** — reuse pattern from `claude-session.js` lines 202-228.
7. **Proxy routing** — steps spawn with `ANTHROPIC_BASE_URL` pointing at proxy for Inspector visibility.
8. **Only two step types** — `condition` (pure JS) and agent (full `claude -p` session).
9. **GUI reuses existing patterns** — `.cap-list`, `.cap-modal`, `.ref-tag`, `.pm-field-row` — no new design language needed.
10. **Test from GUI** — the test modal auto-generates input forms from workflow definition, shows live step-by-step progress with expandable streaming output per step.
