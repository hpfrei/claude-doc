📄 File Operations
Read read-only Read files, images, PDFs, and notebooks from the filesystem ▶

Description

Reads a file from the local filesystem. Supports text files, images (PNG, JPG, etc. — rendered visually), PDFs (up to 20 pages per request), and Jupyter notebooks (.ipynb). Results are returned with line numbers (cat -n format). By default reads up to 2000 lines from the beginning.

Parameters

NameTypeRequiredDescription
file_pathstringrequiredAbsolute path to the file to read
offsetnumberoptionalLine number to start reading from (for large files)
limitnumberoptionalNumber of lines to read (for large files)
pagesstringoptionalPage range for PDFs, e.g. "1-5", "3", "10-20". Required for PDFs over 10 pages. Max 20 pages per request.

Key Behaviors

  • Path must be absolute, not relative
  • Can read any file on the machine
  • Images are presented visually (multimodal)
  • Notebooks return all cells with their outputs
  • Cannot read directories — use ls via Bash for that
Write write Create new files or completely rewrite existing ones ▶

Description

Writes content to a file, overwriting any existing content. For existing files, the file must be read first (via Read) or the tool will error. Prefer Edit for partial modifications — Write is for new files or full rewrites.

Parameters

NameTypeRequiredDescription
file_pathstringrequiredAbsolute path to the file to write
contentstringrequiredThe full content to write to the file

Key Behaviors

  • Overwrites the entire file if it exists
  • Must Read existing files before writing (enforced)
  • Prefer Edit tool for modifications to existing files
Edit write Make targeted string replacements in files ▶

Description

Performs exact string replacements in files. The old_string must be unique in the file, or the edit will fail. Use replace_all to change every occurrence. The file must have been read at least once before editing.

Parameters

NameTypeRequiredDescription
file_pathstringrequiredAbsolute path to the file to modify
old_stringstringrequiredThe exact text to find and replace
new_stringstringrequiredThe replacement text (must differ from old_string)
replace_allbooleanoptionalReplace all occurrences, not just the first unique match. Default: false

Key Behaviors

  • Fails if old_string is not unique (unless replace_all: true)
  • Preserves exact indentation — match whitespace precisely
  • Must read the file first before editing
  • Useful for renaming variables with replace_all

⚙ Execution
Bash exec Execute shell commands and terminal operations ▶

Description

Executes bash commands and returns output. The working directory persists between commands, but shell state (variables, aliases) does not. The shell environment is initialized from the user's profile.

Parameters

NameTypeRequiredDescription
commandstringrequiredThe bash command to execute
descriptionstringrequiredClear, concise description of what the command does
timeoutnumberoptionalTimeout in ms (max 600000 / 10 min). Default: 120000 (2 min)
run_in_backgroundbooleanoptionalRun in background; you'll be notified on completion

Key Behaviors

  • Prefer dedicated tools (Read, Edit, Write, Glob, Grep) over shell equivalents
  • Working directory persists between calls
  • Use && to chain dependent commands; use parallel calls for independent ones
  • Always quote paths with spaces

/update-config skill Configure Claude Code settings, hooks, permissions, and env vars ▶

Description

Configures the Claude Code harness via settings.json. This is the go-to skill for any persistent behavioral configuration: hooks, permissions, environment variables, and settings file modifications.

When Triggered

  • Automated behaviors: "from now on when X", "each time X", "whenever X", "before/after X"
  • Permission changes: "allow npm commands", "add bq permission to global settings", "move permission to user settings"
  • Environment variables: "set DEBUG=true"
  • Hook troubleshooting and any changes to settings.json / settings.local.json

Key Detail

Automated behaviors ("from now on...") require hooks configured in settings.json — the harness executes these, not Claude's memory or preferences. This skill knows how to write those hook configurations correctly.

Example Usage

/update-config allow npm commands
/update-config set DEBUG=true
/update-config when claude stops show a summary

/simplify skill Review changed code for reuse, quality, and efficiency ▶

Description

Reviews recently changed code and checks for opportunities to improve reuse, code quality, and efficiency. If issues are found, it automatically applies fixes. Think of it as a post-implementation cleanup pass.

What It Checks

  • Reuse — Are there existing utilities, helpers, or patterns in the codebase that the new code could leverage instead of reimplementing?
  • Quality — Does the code follow the project's conventions? Are there any anti-patterns, dead code, or unnecessary complexity?
  • Efficiency — Are there performance concerns? Redundant operations? Unnecessary allocations?

When to Use

Run after completing a feature or making significant code changes. Especially useful after large edits spanning multiple files.

Example Usage

/simplify

Hooks are user-configured shell commands that execute automatically in response to Claude Code lifecycle events. They are defined in settings.json (project, user, or global scope) and run by the harness — not by Claude itself. Hooks enable automated behaviors like linting after file writes, notifications on stop, or gate-keeping before tool execution. Use the /update-config skill to set them up.
🔄 Hook Lifecycle
PreToolUse → Tool Executes → PostToolUse → Claude Responds → Stop
Notification → Fires on any notification event
SubagentStop → Fires when a sub-agent completes
PreToolUse hook Runs before a tool is executed — can block or modify the call ▶

Description

Fires immediately before Claude executes any tool call. The hook receives the tool name and input as environment variables / stdin. It can approve, block, or modify the tool call based on its exit code and stdout.

Environment Variables

VariableDescription
CLAUDE_TOOL_NAMEName of the tool about to be called (e.g. Bash, Write)
CLAUDE_TOOL_INPUTJSON string of the tool's input parameters
CLAUDE_SESSION_IDCurrent session identifier

Exit Codes

  • 0 — Allow the tool call to proceed (stdout is shown to Claude as context)
  • 2 — Block the tool call; stdout is passed back as the reason
  • Other non-zero — Hook error; tool call proceeds with a warning

Use Cases

  • Prevent writes to protected files or directories
  • Enforce naming conventions before file creation
  • Require confirmation for destructive Bash commands
  • Log all tool calls for auditing

Example Configuration

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash /scripts/check-safe-command.sh"
      },
      {
        "matcher": "Write",
        "command": "bash /scripts/check-write-path.sh"
      }
    ]
  }
}
PostToolUse hook Runs after a tool completes — react to results ▶

Description

Fires immediately after a tool call completes. Receives the tool name, input, and output. The hook's stdout is injected into the conversation as additional context for Claude to consider alongside the tool result.

Environment Variables

VariableDescription
CLAUDE_TOOL_NAMEName of the tool that was called
CLAUDE_TOOL_INPUTJSON string of the tool's input parameters
CLAUDE_TOOL_OUTPUTJSON string of the tool's output/result
CLAUDE_SESSION_IDCurrent session identifier

Exit Codes

  • 0 — Success; stdout is shown to Claude as additional context
  • Non-zero — Hook error; warning is shown but execution continues

Use Cases

  • Auto-run linter after file writes
  • Run tests automatically after code edits
  • Log tool outputs for compliance
  • Send notifications when specific tools are used
  • Validate generated code against project standards

Example Configuration

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "eslint --fix $CLAUDE_TOOL_INPUT_FILE_PATH"
      },
      {
        "matcher": "Edit",
        "command": "bash /scripts/post-edit-check.sh"
      }
    ]
  }
}

Stop hook Runs when Claude finishes responding — post-turn actions ▶

Description

Fires when Claude completes a turn (stops generating). The hook's stdout is fed back to Claude as a <user-prompt-submit-hook> message, which can prompt Claude to continue or take additional action. If the hook produces output, Claude will process another turn.

Environment Variables

VariableDescription
CLAUDE_STOP_REASONWhy Claude stopped (e.g. end_turn, max_tokens)
CLAUDE_SESSION_IDCurrent session identifier

Exit Codes

  • 0 with stdout — Claude receives the output and continues processing
  • 0 with no stdout — No further action
  • Non-zero — Hook error; warning shown

Use Cases

  • Automatically run tests after Claude finishes writing code
  • Show a summary of all changes made
  • Trigger a build after implementation is complete
  • Chain follow-up prompts based on what Claude did

Example Configuration

{
  "hooks": {
    "Stop": [
      {
        "command": "echo 'Now run the test suite to verify your changes.'"
      }
    ]
  }
}

📝 Configuration Reference
Hooks are defined in settings.json at three scopes (in order of precedence):
  • Project — .claude/settings.json (committed) or .claude/settings.local.json (gitignored)
  • User — ~/.claude/settings.json
  • Global — ~/.claude/settings.global.json (managed by organization)
Hook Object Schema config Structure of a hook entry in settings.json ▶

Schema

{
  "hooks": {
    "<HookType>": [
      {
        "matcher": "<ToolName>",   // optional: filter by tool
        "command": "<shell cmd>",   // the command to run
        "timeout": 10000            // optional: ms (default varies)
      }
    ]
  }
}

Fields

FieldTypeRequiredDescription
matcherstringoptionalTool name to filter on (e.g. "Bash", "Write"). Only applies to PreToolUse and PostToolUse. If omitted, hook fires for all tools.
commandstringrequiredShell command to execute. Has access to environment variables listed per hook type.
timeoutnumberoptionalTimeout in milliseconds. Hook is killed if it exceeds this.
