Tools are Claude's built-in capabilities — the actions the model can request during the agentic loop. Claude emits a structured tool call (name + JSON input), the harness executes it synchronously, and the result is returned into Claude's context. For example, a Bash call sends {"command": "npm test"} and receives stdout/stderr back; a Read call sends a file path and gets the contents.

Permission model: read-only tools (Read, Grep, Glob) need no approval. File-modifying tools (Edit, Write) and Bash require user approval, governed by allow / ask / deny rules evaluated in that priority order. Rules live in settings.json at user, project, or managed scope. The --disallowedTools CLI flag (used by profiles here) adds session-scoped deny rules on top of settings.

Toggle the checkbox on each card to disallow a tool for the current profile. Disabled tools are passed as --disallowedTools flags when spawning claude -p.

Skills are Markdown playbooks that Claude can invoke via the Skill tool. Each skill is a directory containing a SKILL.md file with optional YAML frontmatter and a Markdown body of instructions. Stored in .claude/skills/<name>/SKILL.md (project scope) or ~/.claude/skills/<name>/SKILL.md (user scope).

Two invocation paths: (1) User-invoked — type /skill-name [args] in the prompt. (2) Auto-invoked — Claude reads all skill descriptions loaded into context and calls the Skill tool when the conversation matches. Control this with frontmatter: disable-model-invocation: true hides a skill from Claude's auto-invoke context; user-invocable: false hides it from the / menu.

Execution: skills run synchronously in the main conversation context by default. Set context: fork in frontmatter to run in an isolated sub-agent with no access to conversation history. The $ARGUMENTS placeholder (and positional $0, $1) substitutes user input before Claude sees the content. Frontmatter can also override model, effort, allowed-tools, and scope lifecycle hooks to the skill's lifetime.

Skills supersede the older .claude/commands/ format — a command file and a skill directory with the same name create the same /name slash command, but skills add a directory for supporting files, richer frontmatter, and auto-invocation. Built-in skills (commit, review-pr, etc.) ship with Claude Code and are always available. Deny the Skill tool or specific skills via permission rules to restrict access.

Agents are sub-agent definitions — each one describes a specialized AI instance that runs in its own isolated context window. Stored in .claude/agents/<name>.md (project) or ~/.claude/agents/<name>.md (user), with YAML frontmatter and a Markdown body that becomes the agent's system prompt.

Frontmatter fields: name and description (required), plus optional model, tools (allowlist), disallowedTools (denylist), permissionMode, maxTurns, effort, skills (injected at startup), mcpServers, hooks (scoped to the agent's lifetime), memory, isolation (worktree for git isolation), and background.

Invocation: Claude delegates tasks to agents via the Agent tool, choosing which agent to use based on description matching. Users can also force an agent by name in the prompt, @-mention it, or run a full session as that agent with claude --agent name.

Isolation model: each invocation starts with a fresh context window — the agent sees only its own system prompt and environment info, not the parent conversation history. By default, agents inherit all parent tools; restrict with tools (allowlist) or disallowedTools (denylist). Agents cannot spawn other agents (no nesting).

Return value: when complete, only a summary of the agent's work is returned to the parent context — all intermediate tool calls and verbose output stay in the agent's own context, keeping the main conversation clean.

Sync vs async: foreground agents (default) block the main conversation until complete. Background agents (background: true) run concurrently — the main conversation continues, and the agent's result is delivered later. Built-in agents: Explore (haiku, read-only, fast search), Plan (read-only, architecture), General-purpose (all tools).

Lifecycle hooks in .claude/settings.local.json
Hooks are user-defined automations that fire at specific lifecycle points in Claude Code. Unlike tools (which Claude invokes), hooks are executed by the harness automatically — Claude cannot trigger or bypass them. They live in settings.json (not CLAUDE.md) to prevent prompt injection from modifying hook behavior.

Configuration: three levels of nesting — event name → matcher group (regex filter on tool name, start reason, etc.) → hook handlers. Omit the matcher or use "*" to match all events of that type.

Three handler types: (1) command — runs a shell command; receives JSON context on stdin (session ID, tool name, tool input, tool response); returns decisions via exit code (0 = success, 2 = block) or JSON on stdout. (2) prompt — single-turn LLM evaluation; $ARGUMENTS replaced with the hook's JSON input; returns a yes/no decision. Only for tool events. (3) agent — spawns a sub-agent with tool access (Read, Grep, Glob) to evaluate the situation before returning a decision. Only for tool events.

Execution: synchronous by default — hooks block Claude Code until complete. Set async: true on command hooks for fire-and-forget background execution (logging, notifications). Some events are inherently async and cannot block (Notification, FileChanged, CwdChanged).

JSON output (exit 0): hooks can return {"continue": false, "stopReason": "..."} to halt Claude, {"hookSpecificOutput": {"permissionDecision": "deny"}} to block a tool call (PreToolUse), {"hookSpecificOutput": {"updatedInput": {...}}} to modify a tool's input before it runs, or {"hookSpecificOutput": {"additionalContext": "..."}} to inject context into Claude's view.

Configuration Reference
Hooks live in settings.json at three scopes (loaded in order, later overrides earlier): User (~/.claude/settings.json), Project (.claude/settings.json — committable, shared with team), Local (.claude/settings.local.json — gitignored, personal overrides). Hooks can also be defined in skill and agent frontmatter, scoped to that component's lifetime and auto-cleaned on completion. Managed policy hooks (org-wide) take highest precedence and cannot be overridden.

New Hook

Hook execution

Hooks are run by the harness, not by Claude. They fire automatically at lifecycle events. Synchronous by default — blocks until complete.

Handler types

# command — runs a shell command
# receives JSON on stdin:
# {session_id, tool_name, tool_input,
#  tool_response, cwd, hook_event_name}

# prompt — single-turn LLM evaluation
# $ARGUMENTS replaced with JSON input
# returns yes/no decision

# agent — spawns sub-agent with tools
# (Read, Grep, Glob) to evaluate

Exit codes (command type)

exit 0  # success, parse JSON stdout
exit 2  # block (PreToolUse blocks tool)
other   # non-blocking error, continue

JSON output (exit 0)

{
  "continue": false,
  "stopReason": "message to user",
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "...",
    "updatedInput": {"command": "..."},
    "additionalContext": "injected text"
  }
}

Common patterns

# Auto-format after file writes:
jq -r '.tool_input.file_path //
  .tool_response.filePath' |
  { read -r f; prettier --write "$f"; }
  2>/dev/null || true

# Log all bash commands:
jq -r '.tool_input.command' \
  >> ~/.claude/bash-log.txt

# Block dangerous commands:
jq -r '.tool_input.command' |
  grep -q 'rm -rf' &&
  echo '{"decision":"block"}' ||
  echo '{}'
